# DECISIONS

- 2026-03-14 Codex — Initialized `/doc` tracking files because the repository was missing the required AGENTS.md context structure.
- 2026-03-14 Codex — Treated `rules.md` as the stricter Phase 2 gate while still documenting its conflict with `implementation_blueprint.md`; the audit uses both, but completion calls out the mismatch explicitly instead of silently choosing one.
- 2026-03-14 Codex — Moved transaction ingestion reliability into a Postgres RPC (`public.ingest_transaction`) so transaction creation, queue row creation, and audit insertion happen in one database transaction rather than three separate API calls.
- 2026-03-14 Codex — Added a unique transaction-job index and replay/backfill logic so idempotent retries can repair missing queue rows instead of only returning an existing transaction.
- 2026-03-14 Codex — Kept the worker execution model as an authenticated internal route plus queue claim RPC; scheduled runtime orchestration remains a separate follow-up because deployment steps were explicitly deferred for this task.
- 2026-03-14 Codex — Chose Supabase scheduled Edge Functions as the Phase 2 automation runtime. The hosted `process-jobs` Edge Function currently forwards to the existing Next.js internal worker route using Vault-backed `app_url` and `job_runner_secret` values, which keeps the current worker logic reusable until it is fully migrated off the app runtime.
- 2026-03-14 Codex — Interpreted “Phase 3” according to the repo’s source-of-truth as compliance operations (cases, SAR drafts, KYC), while explicitly documenting that regulatory-document intelligence features from the user prompt are not represented in the current phase plan.
- 2026-03-14 Codex — Prioritized a shared-shell UI refresh over page-by-page redesign so the app could get a visible quality jump quickly. The light/night theme is implemented with a small localStorage-backed `data-theme` toggle instead of adding another theming dependency.
- 2026-03-14 Codex — Introduced `case_notes` as the minimal new schema primitive for Phase 3 so investigator notes and a case timeline can exist without over-designing a larger workflow/event system yet.
- 2026-03-14 Codex — Implemented SAR generation synchronously behind `POST /api/v1/cases/:id/sar-draft` with a deterministic fallback narrative. This keeps Phase 3 functional without adding another queue/worker before the existing job orchestration follow-up is finished.
- 2026-03-14 Codex — Used service-role-backed Phase 3 mutations for multi-table workflows (case creation, alert linking, report generation) while still deriving organization context from the authenticated profile.
- 2026-03-14 Codex — For the rapid MVP pass, prioritized regulatory-document ingestion, AI summarization, search/filtering, and a dashboard-visible intelligence surface over lower-value remaining backlog items such as KYC upload, scheduler activation, and async SAR hardening.
- 2026-03-14 Codex — Implemented regulatory analysis with OpenAI plus a deterministic heuristic fallback so the intelligence workflow remains demonstrable even if API access is unavailable or the model returns invalid output.
- 2026-03-14 Codex — Deferred Supabase env validation to runtime usage for admin/browser clients instead of import/render time. This keeps GitHub Actions and other build-only environments from failing `next build` while preserving explicit runtime errors once an unconfigured auth or worker path is actually used.
